# Effective Requirements Analysis Fixture Source

这是一份用于 KiU fixture 的精简源文档。

## Problem-First Requirements Analysis
需求分析的第一步是确认业务问题，而不是抢着写方案。
用户说“我要一个功能”时，往往只是在表达痛点，不是在给出最终解。
先把目标、场景、失败后果和真实阻力问清楚，后面的方案讨论才有意义。
如果问题没有被定义，页面、接口和报表名称都会把团队带到错误抽象层。
真正稳定的需求描述应当能回答“为什么做、为谁做、解决什么损失”。
这时分析者才适合进入方案比较，而不是被第一个想法绑住。

需求语言如果停留在功能名词，团队就会把模糊前提错误地固化进排期。
先回到问题级需求，是为了给后续拆分、接口和优先级建立稳定边界。

## Business-First Subsystem Decomposition
复杂系统需要拆分，但拆分依据应该是业务职责，而不是技术层。
如果子系统名称只是前端、中台、数据库的重新命名，用户无法确认边界是否完整。
业务视角的拆分更容易让用户参与，也更容易发现遗漏、重叠和跨边界服务关系。
需求分析阶段应优先得到“谁服务谁、谁承担什么业务结果”的结构。
只有这样，模块边界才不会在实现阶段反复漂移。
技术实现当然重要，但它不是需求结构的第一语言。
驾驶、安全、经营、客服这类业务语义，通常比页面、表、接口更适合做第一层切分。

当拆分依据站不住时，后续接口定义也会退化成代码对代码的连接。

## Business Interface Analysis
子系统一旦按业务切开，下一步就要明确它们之间交换的业务价值与责任。
接口分析不只是列字段，而是说明上游输出的是什么业务结果，下游为什么需要它。
一个接口如果只有技术格式，没有业务意义，就无法帮助用户判断完整性。
好的接口描述会同时说明触发条件、责任归属和失败时的影响。
这能把“集成工作”重新拉回业务协同，而不是只剩技术协议。
接口分析因此依赖前面的业务拆分，也反过来验证拆分是否合理。
