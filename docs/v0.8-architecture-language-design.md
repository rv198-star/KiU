# v0.8 架构语言设计

日期：2026-04-26

v0.8 重新梳理 KiU / `学以致用` 的公共叙事：把原书知识转化为可执行的行动力，同时保留原书忠实度、使用边界和用户选择权。

这是一次语言与架构清晰度版本，不回写历史证据，也不改变生成行为。

## 核心架构

公共架构应该像一串动作，而不是一组内部名词：

```text
读准原书 -> 提炼判断 -> 生成技能 / 分流流程 -> 校准应用 -> 验证行动价值
```

| 步骤 | 公共表述 | 精确含义 | 做什么 | 不能做什么 |
| --- | --- | --- | --- | --- |
| 1 | 读准原书 | 保留来源与证据 | 保留原书说了什么、在哪里说、证据如何锚定。 | 不能在尚未形成可行动判断前，就把原书材料当成行动建议。 |
| 2 | 提炼判断 | 从证据中提炼可迁移判断 | 从原书证据中提炼能迁移到其他情境的判断。 | 不能把书降格为摘要、摘录或百科问答。 |
| 3A | 生成技能 | 发布行动技能 | 发布能帮助用户判断、取舍、拒绝、行动或复盘的技能。 | 不能把章节标题、练习题、概念容器或确定性流程发布成厚技能。 |
| 3B | 分流流程 | 流程单独路由 | 确定性步骤保留为可审计流程工件，并在合适时把用户路由过去。 | 不能把判断型技能压成清单，也不能把清单膨胀成技能。 |
| 4 | 校准应用 | 隔离地加入应用语境 | 按需加入当前事实、风险提示、使用限制和安全门。 | 不能把原书忠实技能改写成混合现实语境的产物。 |
| 5 | 验证行动价值 | 检查产物是否真的有用 | 检查产物是否帮助用户发现信号、定义问题、解决行动、复盘校准。 | 不能用内部证据冒充真实用户或外部验证。 |

## 核心公共表达

`技能不是摘要`：KiU 不是为了总结书，而是把原书知识转化成有边界、可触发的行动力。

`学到最后要能用`：学习的终点不是“知道”，而是能改善具体判断、选择、行动、拒绝和复盘。

这两个表达是 v0.8 的公共记忆点。`用而不染` 保留为“校准应用”的架构原则：应用语境可以帮助使用技能，但不能污染或改写原书技能。

## 覆盖性审查

| 当前能力 | v0.8 架构归属 | 覆盖判断 |
| --- | --- | --- |
| 来源 bundle、证据锚点、来源追溯、来源快照 | 读准原书 | 已覆盖。 |
| 候选抽取、提炼、seed 校验、refinement loop | 提炼判断 | 已覆盖。 |
| 已发布的 `bundle/skills/*/SKILL.md` | 生成技能 | 已覆盖。 |
| workflow candidate、gateway/router artifact | 分流流程 | 已覆盖。 |
| 当前语境、事实核验、幻觉门、应用限制 | 校准应用 | 已覆盖。 |
| 方法检查、来源/事实检查、行动价值分、用户侧评分卡 | 验证行动价值 | 已覆盖。 |
| 参照对比、盲评包 | 验证行动价值中的证据等级 | 作为证据等级覆盖，不作为核心架构层。 |

覆盖结论：这套动作链可以覆盖当前 KiU 行为，不需要在主解释中依赖旧版本术语或外部参照项目术语。

## 清晰度审查

| 检查项 | 结果 |
| --- | --- |
| 新读者能否在 90 秒内说明项目？ | 可以：KiU 把书转化为可执行的行动力，而不是摘要。 |
| 是否过度造词？ | 没有：五个动作短语，加两个核心公共表达。 |
| 是否依赖旧版本或外部项目术语？ | 不依赖：历史词汇进入归因说明、历史证据和 glossary。 |
| 是否保留工程边界？ | 保留：来源忠实、流程分流、应用隔离都明确存在。 |
| 是否能支撑 README？ | 可以：README 可以按问题、架构、证据边界、使用方式展开。 |

清晰度结论：公共文档应优先使用动作短语。更精确的内部名词只在必要时补充。

## 中文命名审查

中文命名要单独审查。一个词可以架构上准确，但如果第一次读的人看不懂，就不适合作为公共语言。

| 早期工作名 | 问题 | v0.8 公共表述 | 决定 |
| --- | --- | --- | --- |
| 原书证据层 | `证据层` 准确但偏法律化、抽象。 | 读准原书 | README 和叙事文档使用公共表述，技术文档再谈证据。 |
| 判断提炼层 | 准确但可能被误解成摘要提炼。 | 提炼判断 | 用动宾结构强调产出判断，而不是产出摘要。 |
| 行动技能层 | 基本清楚，但可能被误解为只做直接执行。 | 生成技能 | 架构流中用 `生成技能`，并解释技能包括判断、拒绝、复盘。 |
| 流程路由层 | 合理但偏内部。 | 分流流程 | 使用更直白的公共表述；技术文档可保留 workflow routing 作为英文别名。 |
| 应用校准层 | 比更宽泛的对齐说法好，但仍偏抽象。 | 校准应用 | 用动宾结构，并把 `用而不染` 作为该步骤边界。 |
| 行动价值验收 | 对 gate 准确，但公共文档太硬。 | 验证行动价值 | 公共文档用 `验证行动价值`；release gate 和 scorecard 可保留 `行动价值验收`。 |

中文命名标准：

- 一个术语最多用一句短句就能解释。
- 不要求读者了解 KiU 历史版本或外部参照项目。
- 能作为 README 标题，而不只适合工程报告。
- 如果精确名太生硬，公共文档使用更白话的动作短语，技术文档记录精确别名。

命名结论：v0.8 公共语言优先使用动作短语，少用名词化层级名。

## v0.8 之后的语言规则

- 新主线文档优先使用：读准原书、提炼判断、生成技能、分流流程、校准应用、验证行动价值。
- 历史术语只出现在历史证据、归因说明或 glossary 中。
- 新说明文档如果必须提历史术语，只首次说明其历史含义，后文继续使用 v0.8 术语。
- 不为了改名批量改写旧报告和旧计划。
- 不允许用语言简化削弱证据诚实或流程/判断边界。
- README 应先使用公共动作短语，再在读者理解链路后补充精确工程别名。
