# v0.8.3 Skills Distribution Design

## Position

v0.8.3 is a distribution and installability release. It does not change KiU's generation quality model, source extraction model, or value-gain pressure-chain renderer.

The release answers one missing question: if KiU is a SKILLS project, how can KiU itself and KiU-generated skills be installed and used as skills without cloning the full engineering repository or reading the review pack manually?

## Core Decision

KiU needs two installable layers:

1. `skills/kiu`: KiU itself as an installable project skill.
2. `installable-skills/`: a generated installation view for skills produced by KiU.

These layers are intentionally separate. `skills/kiu` teaches an agent how to use KiU. `installable-skills/` contains book-derived skills that can be installed independently.

## Current Mechanism Assumption

The current Codex skills mechanism treats a skill as a directory containing `SKILL.md` with YAML frontmatter. Optional subdirectories such as `references/`, `scripts/`, and `assets/` may be included. GitHub-path installation is supported by the Codex skill installer into `$CODEX_HOME/skills/<skill-name>`.

v0.8.3 should not depend on official curated marketplace acceptance. Official listing can be considered later, but it is external, slower, and should not block this release.

## Layer 1: KiU Project Skill

Create:

```text
skills/kiu/
  SKILL.md
  references/
    project-boundaries.md
    generation-workflow.md
    distribution-workflow.md
```

`skills/kiu/SKILL.md` should be concise. It should explain that KiU means `Knowledge in Use`, Chinese name `学以致用`, and that KiU turns source books into bounded, reviewable action skills.

It should tell the agent when to use this skill:

- when the user asks what KiU is;
- when the user wants to inspect KiU review packs;
- when the user wants to generate skills from source material;
- when the user wants to export or install KiU-generated skills;
- when the user needs KiU evidence-boundary discipline.

It should not include long release history or generated book content. Detailed instructions should live under `references/`.

Expected install form:

```text
$skill-installer install https://github.com/rv198-star/KiU/tree/v0.8.3/skills/kiu
```

## Layer 2: Installable Generated Skills

`review-pack/current` remains an audit and review view. It must not become the installation view.

Create a generated install view:

```text
installable-skills/
  manifest.json
  kiu-financial-statement-accounting-quality-signal-check/
    SKILL.md
    references/provenance.md
  kiu-financial-statement-business-value-anchor-check/
    SKILL.md
    references/provenance.md
  ...
```

The directory name must be globally unique:

```text
kiu-<source-id>-<skill-id>
```

This prevents collisions such as two books both generating `workflow-gateway` or `role-boundary-before-action`.

The exported `SKILL.md` should be directly installable. It may preserve the generated skill body, but it must have install-safe frontmatter:

```yaml
---
name: kiu-shiji-role-boundary-before-action
description: Use when applying a Shiji-derived role-boundary judgment skill to decide whether to act, defer, or refuse a requested action.
---
```

Source, run, evidence, and review-pack links should be moved into `references/provenance.md` or `manifest.json` rather than bloating the top of `SKILL.md`.

## Exporter

Add a deterministic exporter:

```text
scripts/export_installable_skills.py
```

Inputs:

- `--review-pack review-pack/current`
- `--output installable-skills`
- optional `--clean`

Responsibilities:

- read `review-pack/current/manifest.json`;
- find each `books/<book-id>/generated-skills/<skill-id>/SKILL.md`;
- write a globally unique installable skill directory;
- rewrite or wrap frontmatter to use the unique install name;
- write `references/provenance.md`;
- write top-level `installable-skills/manifest.json`;
- refuse duplicate output names;
- refuse missing `SKILL.md` files;
- preserve generated value-gain markers.

## Local Installer Smoke

Add a minimal local installer smoke path rather than a full package manager.

For v0.8.3 it is enough to test that the generated installable directories can be copied into a temporary skills home:

```text
/tmp/kiu-install-smoke/skills/<skill-name>/SKILL.md
```

This proves the generated structure is compatible with the local skills directory layout. It does not claim official marketplace listing or multi-agent compatibility.

## Documentation

README should include two installation paths:

1. Install KiU itself:

```text
$skill-installer install https://github.com/rv198-star/KiU/tree/v0.8.3/skills/kiu
```

2. Install KiU-generated skills:

```text
$skill-installer install https://github.com/rv198-star/KiU/tree/v0.8.3/installable-skills/<skill-name>
```

For bulk local installation, README can point to the smoke-compatible script if implemented, but bulk installation is secondary to GitHub-path installability.

## Release Gate

v0.8.3 passes when:

- `skills/kiu/SKILL.md` exists and has valid frontmatter.
- `installable-skills/manifest.json` exists.
- the current 21 repository-visible generated skills export into 21 globally unique installable skill directories.
- every installable skill contains `SKILL.md` with valid `name` and `description` frontmatter.
- every installable skill preserves the v0.8.2 value-gain markers.
- a temporary local skills-home smoke test can copy and discover 22 skills: 1 KiU project skill plus 21 generated skills.
- README explains install commands and evidence boundaries.
- CI validates the installable structure.

## Non-goals

- Do not pursue official curated marketplace acceptance in v0.8.3.
- Do not redesign the generation pipeline.
- Do not edit generated skill content manually to improve quality.
- Do not make `review-pack/current` double as the install package.
- Do not build a full package manager or remote registry.
