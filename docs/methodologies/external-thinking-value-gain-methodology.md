# 外部工具：模块价值增益法

## 定位

模块价值增益法来自外部工具包 `thinking-value-gain-methodology-v0.1`。KiU 只把它作为外部工具方法论调用，用于已经成形但可能价值偏薄的具体模块审计。

它不是 KiU 原生方法论，不进入项目公开主叙事，也不能替代 `学以致用方法论`、三层递归五步法、极限演绎与场景投影法或系统效率原则。

一句话定位：当一个模块看起来已经完整，但仍可能没有给决策、行动、复审、交接或复用带来足够价值时，用它做有限轮增益审计。

## 触发条件

满足以下情况时可以调用：

- 产物结构已经存在，但判断力、边界、取舍或使用说明偏薄。
- 自动评分通过，但人工读起来仍像模板化完成品。
- 下游使用者还需要自己补关键判断，才能把产物真正用起来。
- 某个模块需要被交接、复审、复用或作为后续生成编排依据。

不满足以下情况时不应调用：

- 只是想把文本写得更长。
- 试图用外部方法论替代原书证据。
- 试图直接补写已经生成的 source-derived `SKILL.md` 正文。
- 试图用单个样本证明跨书普适有效。

## KiU 内部用法

正确用法是把它放在生成编排或审计层：

1. 先确定待审计模块，例如某本书的技能契约、触发边界、使用说明或评估证据。
2. 判断该模块缺的是哪类价值：决策价值、证据价值、交接价值、风险降低价值、复用价值或执行价值。
3. 做有限轮审计，输出增益建议、返工项或冻结理由。
4. 如果建议会改变生成物内容，必须回到生成链路、结构化文件或模板层修改，不能只手工改发布产物。
5. 完成后重新跑对应评分、结构一致性检查和必要的人工复核。

## 禁止用法

- 禁止把外部方法论文本粘进原书派生的 `SKILL.md`，因为这会污染来源忠实性。
- 禁止把它当成真实用户验证、外部盲评或项目方向证据。
- 禁止只修改 Markdown 展示层，而不同步与之耦合的结构化数据。
- 禁止把 Financial Statement 单书实验说成五本书普遍有效。

## 已完成试验

截至 2026-04-26，只完成过 Financial Statement 单书 A/B 初测。

结果摘要：

- A 组是当前基线产物。
- B 组尝试在技能文本中补充价值增益表达，但触碰了与结构化摘要耦合的展示段，暴露出“只改 Markdown 会造成产物漂移”的风险。
- B2 组只补充契约和理由层的价值检查，不改评估摘要；自动评分与 A 组基本持平，但人工阅读上更容易看出下游使用价值。

这个试验只证明该工具适合做模块审计和生成编排提示，不证明它已经带来跨书稳定增益。

## 五本齐发验证门槛

如果后续要宣称该工具对 KiU 有普遍价值，必须至少完成五本书 A/B 验证：

- 对每本书分别生成不使用该方法的 A 组和使用该方法编排的 B 组。
- B 组必须通过生成链路或模板层实现，不能手工改最终产物。
- 比较三类结果：自动评分是否退化、人工读物是否更好用、来源忠实和边界是否被污染。
- 五本中至少多数书出现清晰正向收益，且不能有严重来源污染或结构一致性失败，才允许升级为稳定调用规范。

在五本验证完成前，文档中只能称它为“外部可选审计工具”。
